1. Introduction

DCL files, text files referenced by nX-U8/100 core development support software, contain the device-specific particulars for the corresponding microcontroller.

MACU8 assembler package components can therefore tailor their operation to the target device by reading in the contents of the appropriate DCL file, specified in the source code for the user program.

Only the assembler (RASU8) reads in the DCL file. It then stores the necessary DCL file information in the object files that it creates for use by the linker (RLU8) and the symbolic debugger.

This document gives the convention for naming DCL files and describes the declaration syntax specifying the following microcontroller particulars in them.

(1) core identifier
(2) microcontroller name
(3) code memory space and special regions therein
(4) data memory space and special regions therein
(5) SFR access
(6) reserved words expressing addresses
(7) hardware interrupt addresses
(8) instructions available

This information is also necessary to write user programs. This document therefore helps the developer read it from the DCL file for the target device.

[Note]
Never modify the contents of a DCL file. Doing so can invalidate operation of the software referencing that file.

2. DCL file names

When using the MACU8 assembler package, you must always specify the name of the DCL file containing the device-specific particulars for the target microcontroller. DCL files use the following naming convention.

The DCL file name is the microcontroller name with M replacing the ML portion and the file extension ".DCL".

<Examples>

Microcontroller Name

DCL File Name

ML610101

M610101.DCL

ML610201

M610201.DCL

 

The source code for the user program specifies the DCL file name with the TYPE directive. This directive has a single operand, the base name of the DCL file.

The assembler then reads in the corresponding DCL file.

<Example 1> TYPE(M610101)

<Example 2> TYPE(M610201)

3. DCL file contents

Now let us turn to the contents of DCL files. A DCL file consists of DCL instructions giving the device-specific particulars for the target microcontroller. The descriptions below give the DCL instruction syntax following by a detailed description of that instruction.

 

A lower case string in a syntax summary indicates an operand for that DCL instruction. The meaning of that operand appears in the description following the syntax summary.

 

An operand representing an address or other numerical value can be an integer constant, symbol, or expression.

 

Some syntax summaries use the following notation.

 

Notation

Meaning

...

Repeat the immediately preceding item as necessary

 

The following describes the microcontroller particulars from DCL files of particular importance to developers using the MACU8 assembler package.

 

The DCL instructions not described here are used internally by the MACU8 assembler package.

 

3.1 Core identifier

The DCL file specifies the core architecture for the target microcontroller with an identifier using the following declaration syntax.

 

#CORE "cpu-core"

 

cpu-core specifies the core type for the target microcontroller. The following string is available.

 

nX-U8/100

 

3.2 Microcontroller name

The following DCL instruction specifies the name for the target microcontroller.

 

#TARGET "microcontroller-name"

 

microcontroller-name is the name for the target microcontroller. This name distinguishes microcontrollers which has nX-U8/100 core.

 

3.3 Code memory space physical segment #0 and special regions therein

The following DCL instruction specifies the range of addresses available in the code memory space.

 

#ROM rommin , rommax

 

rommin and rommax are the offsets, respectively, for the start and end of code memory addresses available in physical segment #0.

 

Immediately following this DCL instruction are ones giving the address ranges for special areas in physical segment #0.

 

The following DCL instruction specifies the address range for the interrupt vector table.

 

SWI_VECTOR min_address , max_address

 

The following DCL instructions specify address ranges available for allocating segments of the corresponding usage type to the code memory space.

 

CODE min-address , max-address

...

 

NVRAM min_address , max_address

...

 

CODE indicates an address range containing ROM; NVRAM, one containing nonvolatile memory.

 

3.4 Data memory space physical segment #0 and special regions therein

The following DCL instruction specifies the range of addresses available in the data memory space.

 

#RAM rammin , rammax

 

rammin and rammax are the offsets, respectively, for the start and end of data memory addresses available in physical segment #0.

 

Immediately following this DCL instruction are ones giving the address ranges for special areas in physical segment #0.

 

The following DCL instruction specifies address ranges available for allocating segments of the corresponding usage type to the data memory space.

 

DATA min-address , max-address

...

 

The following DCL instruction specifies the SFR area.

 

SFR min-address , max-address

 

The following DCL instruction specifies the internal RAM area.

 

INTERNALRAM min-address , max-address

 

The following DCL instruction specifies the NVRAM area. NVRAM min-address, max-address

 

NVRAM min-address , max-address

 

The following DCL instruction specifies a combination of starting and end addresses available for use as a ROM window.

 

ROMWINDOW min-address , max-address , winval

 

min-address and max-address are the offsets, respectively, for the start and end of the ROM window. winval is the actual value used by the software to select that combination.

 

Zeros for both min-address and max-address indicates that the software has the option of not using the ROM window function.

 

3.5 Address ranges in physical segments #1 and higher

The following DCL instruction specifies the memory space contents in physical segments #1 and higher.

 

#EXTEND num_of_romseg, num_of_ramseg

 

num_of_romseg specifies the number of code segments available; num_of_ramseg, the number of data segments. If num_of_romseg is 10, for example, physical segments #0 through #9 contain code memory.

 

Immediately following this DCL instruction are ones giving address ranges of the corresponding usage types in physical segments #1 and higher.

 

The following DCL instruction specifies a RAM address region.

 

RAM min-address , max-address

...

 

The following DCL instruction specifies a ROM address region.

 

ROM min-address , max-address

...

 

The following DCL instruction specifies an NVRAM address region.

 

NVRAM min-address , max-address

...

 

 

3.6 SFR access

The SFR area contains addresses mapped to hardware registers, which differ from regular memory in sometimes limiting read/write access. The following DCL instructions specify whether particular byte addresses support such access with data widths word, byte, or bit. The assembler uses this information to check the validity of all program accesses to addresses in the SFR area.

 

#ACCESS

address : length read_bit write_bit

...

 

address is an address in the SFR area.

 

length is a field indicating read/write access for data widths word and byte.

The four characters in this field have the following meanings.

 

(1) The first character indicates whether read access is available at the word level.

 

Character

Meaning

R

Read access available at word level

-

Read access not available at word level

S

Read access available at word level, upper byte of word-sized SFR

 

 

(2) The second character indicates whether write access is available at the word level.

 

Character

Meaning

W

Write access available at word level

-

Write access not available at word level

S

Write access available at word level, upper byte of word-sized SFR

 

 

(3) The third character indicates whether read access is available at the byte level.

 

Character

Meaning

R

Read access available at byte level

-

Read access not available at byte level

 

 

(4) The fourth character indicates whether write access is available at the byte level.

 

Character

Meaning

W

Write access available at byte level

-

Write access not available at byte level

 

 

read_bit is a numerical value whose binary equivalent represents a mask indicating whether read access is available for the corresponding bits: "1" for yes; "0" for no. A hyphen in this position indicates that no bit supports read access.

 

read_bit is a numerical value whose binary equivalent represents a mask indicating whether read access is available for the corresponding bits: "1" for yes; "0" for no. A hyphen in this position indicates that no bit supports read access.

 

<Example> #ACCESS

P0 : --RW 1111_1111B 1111_1111B

 

3.7 Reserved words expressing addresses

This section of the DCL file defines reserved word symbols expressing addresses for use instead of numerical addresses by user application programs. Examples include names for SFRs.

 

These reserved word symbols expressing addresses are grouped by usage type, so the rest of this section describes the DCL instruction for each type.

 

3.7.1 Reserved words expressing code addresses

 

#DEFCODE

symbol address

...

 

This instruction defines a reserved word symbol of usage type CODE for the specified address.

 

3.7.2 Reserved words expressing data addresses

 

#DEFDATA

symbol address

...

 

This instruction defines a reserved word symbol of usage type DATA for the specified address.

 

3.7.3 Reserved words expressing bit addresses

 

#DEFBIT

symbol address

...

 

This instruction defines a reserved word symbol of usage type BIT for the specified address.

 

3.8 Microcontroller instructions

The following DCL instruction defines the microcontroller instructions available to user application programs.

 

3.8.1 Declaration of reserved instructions

 

#INSTRUCTION

instruction ...

...

 

instruction represents the mnemonic for such an instruction.

 

3.8.2 Declaration of using coprocessor register

 

#COPROCESSOR

 

When this DCL instruction is specified, the following coprocessor registers are available.

 

CR0 CR1 CR2 CR3 CR4 CR5 CR6 CR7

CR8 CR9 CR10 CR11 CR12 CR13 CR14 CR15

CER0 CER2 CER4 CER6 CER8 CER10 CER12 CER14

CXR0 CXR4 CXR8 CXR12

CQR0 CQR8

 

3.9 Declaration of information associated with interrupt

#INTERRUPT

VectorSymbol EnableFlag RequestFlag

...

 

This instruction is referenced in simulation mode in the DTU8 debugger. The interrupt enable flag (EnableFlag) and interrupt request flag (RequestFlag) that are associated with the interrupt vector (VectorSymbol) are described.
