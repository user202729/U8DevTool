2. Updated Contents

The following updates have been modified in this release:

  • CCU8 V3.41

(1)   The feature which optimizes register saving and restoring for the function entrance and exit has been added.

This feature is controllable by a command line option (/Orp, /Orpn) and OPTIMIZATION pragma.

Please specify a /Orpn option, when you output the same code as usual.

For details, please refer to "3.2.3 Optimization option" and "5.9.1 OPTIMIZATION pragma" of "CCU8 user manual".

 

(2)   A part of restriction of INLINE pragma has been relieved.

Accordingly, a function including branch (for example, if sentence, for sentence, etc.) is the candidate of an inline expansion.

The overhead of a function call can be reduced by specifying by INLINE pragma to the function called only once.

For this reason, code size is reducible.

For details, please refer to "5.3 INLINE pragma" of "CCU8 user manual".

*: When a function is specified by INLINE pragma, the inline expansion of the function by which an inline expansion was not carried out in the compiler before V3.41 may be carried out and generate the different code in V3.41.

 

(3)   The problem which an internal error issues has been fixed to description which makes a variable declaration after that with reference to a constant within an inline function.

[Occurrence Conditions]

In the earlier version than CCU8 V3.41, this problem occurs when the following condition are satisfied.

(a) The optimization option (/ O m, / O t, / O g, either) is specified at the compile time.

(b) There is an expression which refers to a constant within the function specified by inline pragma.

(c) The constant value which added 1 to the constant expression of (b) is used by the variable declaration after the function definition of (b).

(d) Other function definitions do not exist between (b) and (c).

[Occurrence Example]

#pragma inline inlinefunc

void inlinefunc( void )

{

volatile char loc;

loc = 4; // occurrence condition (b)

}

 

char tbl = 5; // occurrence condition (c)

4. Restrictions

The current version of the CCU8 compiler has the following restrictions:

(1) The code generation to access of the local variable assigned to the stack within the inline function which has a nest relation may be wrong.

[Occurrence Conditions]

This problem occurs when the following conditions are satisfied.

(a) The nesting inline functions exist.

Here, the inline function by the side of a call is defined as inlinefunc1. And the inline function called from inlinefunc1 is defined as inlinefunc2.

(b) The call for inlinefunc2 satisfies one of the following conditions.

(b-1) The call of inlinefunc2 exists in the loop of inlinefunc1.

(b-2) There are two calls or more from inlinefunc1 to inlinefunc2 which does not have return value.

(c) The description in the inline function satisfies one of the following conditions.

(c-1) Two or more volatile local variables exist in inlinefunc1. And the local variables are specified as two or more arguments of inlinefunc2 call, and the arguments are accessed in inlinefunc2.

(c-2) The volatile local variable of inlinefunc1 is specified as one or more arguments of inlinefunc2 call. And, the argument and one or more volatile local variables of inlinefunc2 are accessed in inlinefunc2.

(c-3) One or more volatile local variables of inlinefunc1 are accessed before the inlinefunc2 call in the same block. And one or more volatile local variables of inlinefunc2 are accessed in inlinefunc2.

(d) The optimization option is specified at the compile time. In IDEU8, it corresponds to one of the following options. The inside of a parenthesis indicates command line option.

- Basic (No option)

- Loop optimization only (/ O l)

- Global optimization only (/ O g)

- Both loop and global optimization (/ O g / O l)

- Maximum possible optimization (/ O m)

- Optimization speed (/ O t)

[Occurrence Example]

#pragma inline inlinefunc1

#pragma inline inlinefunc2

 

int gvar1;

 

void inlinefunc2( int a, int b )

{

int lVar;

lVar = a + b;

gvar1 += lVar;

}

{

volatile int b = 40;

int index;

 

index = 0;

for(index =0; index <= 1; index++)

{

inlinefunc2( a, b );

}

}

 

void func(void)

{

inlinefunc1();

}

[Workaround]

Please remove on of inline pragma.

 

//#pragma inline inlinefunc1 // remove inline pragma for inlinefunc1

#pragma inline inlinefunc2

 

int gvar1;

 

void inlinefunc2( int a, int b )

{

...

}

 

void inlinefunc1(void)

{

...

for(index =0; index <= 1; index++)

{

inlinefunc2( a, b );

}

}

(4) An application error may occur to description containing shift operation and a function call.

[Occurrence Conditions]

This problem occurs when the following conditions are satisfied.

(a) The optimization option (/ O m, / O t, / O g, either) is specified at the compile time.

(b) There is an expression which assigns the result of the AND(&) operation of a variable and a constant for a variable.

(c) There is shift operation following (b). And the bit width of the shift operation is a variable after assignment of (b).

(d) There is function call following (c).

(e) The next of (d), there is an expression which assigns the result of the AND(&) operation of a variable and a constant for a variable.

(f) The next of (e), There is shift operation following (e). And the bit width of the shift operation is a variable after assignment of (e).

(g) The constant used by the AND(&) operation of (b) and (e) is the same value.

 

 

[Occurrence Example]

tmp = (tmp2 << bit_pos);

fn();

bit_pos = arg1 & 0x0f;

tmp = (0x01 >> bit_pos);

}

 

[Workaround]

Specify optimization pragma with default to the function which a problem occurs. An example is shown below.

{

...

}

#pragma optimization

 

(5) W5013 may not be outputted to the part which should output warning W5013.

[Occurrence Conditions]

This problem occurs when the following condition (a) or (b) is satisfied.

(a) The char type or unsigned char type value is specified to the argument of int, unsigned int, short or unsigned short type.

(b) The int, unsigned int, short or unsigned short type value is specified to the argument of char or unsigned char type.

 

[Occurrence Example]

void func_uc(unsigned char x);

void main(void) {

func_uc(0x100U); ...(A)

}

Warning W5013 means that conversion of a type was performed. In the above (A), the constant 0x100U is unsigned int type. And the constant value is converted to unsigned char type. So W5013 should be issued, but it does not issue.

[Workaround]

There is no workaround. Please care about that this warning may not be issued.

[Occurrence Conditions]

This problem occurs when the following conditions are satisfied.

(a) The optimization option (/ O m, / O t, / O g, default either) is specified at the compile time.

(b) The common expression which performs logical OR operation is described in the inside of if-else block and the behind.

(c) The common expressions of logical OR operation are the same of (b).

(d) The value decided to values other than zero at the time of compile is specified as the operand of Logical OR.

(In the following example, the variable Operand corresponds)

 

[Occurrence Example]

int Test(unsigned int ResultVar, int CondnVar)

{

unsigned int Operand = 1;

if (10 > CondnVar)

{

ResultVar = ResultVar || Operand;

}

else

{

ResultVar = ResultVar || Operand;

}

ResultVar = ResultVar || Operand;

return ResultVar;

}

 

[Workaround]

Declare with volatile the variable used in a common expression.

volatile unsigned int Operand = 1;

 

 

(7) It is possible that the DT8 debugger cannot reference variables correctly because of the effect by compiler optimization.

If, during the course of CCU8 compiler optimization, a local variable is assigned to a register, the debugging information consistency may be lost; as a result, it becomes impossible for the DTU8 debugger to reference variables correctly.

 

 

(8) The address of the function argument assigned to the register cannot get.

When the function argument is assigned to the register, the address of the argument cannot get.

int *p;

void func(int arg1) <== The argument arg1 is assigned to the register.

{

p = &arg1; <== For this line, CCU8 issues an error.

(E5009 : '&' on register variable)

}

If it needs to get the address of the function argument, then qualify the function using __noreg keyword.

By qualifying with __noreg the function, the all function arguments are assigned to a stack, and can get the address of the argument.

void __noreg func(int arg1)

{

p = &arg1;

}
