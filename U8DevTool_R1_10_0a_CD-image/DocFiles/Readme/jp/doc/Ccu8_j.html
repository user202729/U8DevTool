2. 更新内容

今回のリリースでは、以下の内容が更新されています。

  • CCU8 V3.41

(1)  関数入口・出口のレジスタ退避・復帰を共通化する機能を追加しました。

関数入口・出口のレジスタ退避・復帰の共通化を行うことにより、コード削減を実施できます。

この機能は、コマンドラインオプション（/Orp, /Orpn）、およびOPTIMIZATIONプラグマにて制御できます。

従来と同様のコードを出力したい場合には、/Orpnオプションを指定してください。

詳細については、『CCU8ユーザーズマニュアル』の「3.2.3 最適化オプション」、「5.9.1 OPTIMIZATIONプラグマ」を参照してください。

 

(2)  INLINEプラグマの制限を緩和しました。

INLINEプラグマの制限の一部を緩和することにより、if文やfor文などの分岐を含む関数もインライン展開できるようにしました。

一度しか呼ばれない関数をINLINEプラグマで指定することにより、関数呼び出しのオーバーヘッドを減らすことができるため、コード削減を実施できます。

詳細については、『CCU8ユーザーズマニュアル』の「5.3 INLINEプラグマ」を参照してください。

※関数をINLINEプラグマで指定していた場合、V3.41よりも前のバージョンではインライン展開されなかった関数が、V3.41ではインライン展開され異なるコードが生成される場合があります。

 

(3)  インライン関数にて定数を参照し、その後に変数宣言をするとインターナルエラーが発生する問題を改修しました。

[発生条件]

V3.41よりも前のバージョンでは、以下の条件を満たす場合に本不具合が発生します。

①  コンパイラの最適化オプションに、(/Og, /Om, /Ot)のいずれかを指定している。

IDEU8では、[コンパイル/アセンブル オプション]の[最適化]タブの以下のいずれかに該当します。

・広域的な最適化を行い、ループに関する最適化を行わない

・広域的な最適化とループに関する最適化を行う

・最適化を繰り返して、最大限に最適化する

・実行速度の最適化

 

② inlineプラグマで指定された関数内にて、定数を参照する式がある。

③ ②の定数式に1を加算した定数値が、②の関数定義の後で、関数外の変数宣言で使用されている。

④ ②と③の間に、他の関数定義がない。

 

[発生例]

#pragma inline inlinefunc

void inlinefunc( void )

{

    volatile char loc;

    loc = 4;        // 条件②

}

 

char tbl = 5;      // 条件③

 

 

以降は過去のバージョンの更新内容です。

  • CCU8 V3.31

(1)  char型メンバのみを持つ構造体が、入れ子の構造体メンバとして宣言され、その構造体メンバにアクセスする記述に対し、/Zpオプションを指定してコンパイルすると不正なコードを生成する問題を改修しました。

[発生条件]

V3.31よりも前のバージョンでは、以下の条件を満たす場合に本不具合が発生します。

①  コンパイル時に/Zpオプションを指定している。

② char型メンバのみを持つ構造体がある。

③上記②の構造体をメンバとする構造体があり、入れ子の構造体メンバが奇数番地に割り付けられる

④上記③の構造体メンバに対し、間接的にアクセスする記述がある。または、上記③の構造体メンバに上記②の関数戻り値の構造体を代入している。

 

[発生例]

typedef struct {

    unsigned char   a;

    unsigned char   b;

} STR1 ;

 

typedef struct {

    unsigned char   a;

    STR1            b;

} STR2 ;

 

STR1    str1 = { 10, 20 };

 

void sub(STR1 *p)

{

    *p = str1;  <== ワードバウンダリに違反するコードを生成する

}

 

void main(void)

{

    STR2    str2;

    sub(&str2.b); <== str2.bのアドレスは奇数番地となる

}

 

 

(2)  char型メンバのみを持つ構造体に対しポインタを介してアクセスする記述に対し、/Zpオプションを指定してコンパイルすると不正なコードを生成する問題を改修しました。

[発生条件]

V3.31よりも前のバージョンでは、以下の条件を満たす場合に本不具合が発生します。

① コンパイル時に/Zpオプションを指定している

② char型メンバのみを持つ構造体があり、構造体のサイズが3バイトである。

③上記②の構造体型のポインタを介してアクセスする記述を行っている。

 

[発生例]

typedef struct {

    unsigned char a, b, c ;

} STR ;

STR w[3];

STR d2[3] = { { 0, 1, 2}, { 3, 4, 5}, { 6, 7, 8} };

 

void sub( int x, STR *p )

{

    *p = d2[x] ;  <== ポインタpを介してw[1]にアクセスする際、ワードバウンダリに違反するコード

                      を生成する

}

 

void main(void)

{

    sub(1,&w[1]);  <== w[1]のアドレスは奇数番地となる

}

(4)  ループ内での後置インクリメントの記述に対して、誤ったコードを生成する問題を改修しました。

[発生条件]

V3.21よりも前のバージョンでは、以下の条件を満たす場合に本不具合が発生します。

① 変数VAR1、VAR2の型が同じ基本型(char, short, int, long)で同じ符号(unsigned/ signed) である。

② VAR1はループ制御変数である。

③ VAR1とVAR2は同じ値で初期化されている。

④ VAR1とVAR2はループ内にて同じ値で加算または減算される。

⑤ VAR1とVAR2はループ内で1度だけ値が定義される。

⑥ VAR1がループ内で定義される前にVAR2 が使用される。

 

[発生例]

int res;

int main( void )

{

    int a = 0 ;

    int i ;

 

    for (i = 0; i < 1; i++) {

        a++ ;

        if ( a != 1 ) {

            res = 1;

        } else {

            res = 0;

        }

    }

 

    return 0 ;

}

(6)  アドレスとオフセットを用いて間接コールを行う記述に対し、誤ったコードを生成する問題を改修しました。

[発生条件]

V3.21よりも前のバージョンでは、以下の条件を満たす場合に本不具合が発生します。

① typedefを用いて関数ポインタ (FUNCTION_PTR) が宣言されている。

(7)  構造体または共用体の配列をメンバに持つ構造体/共用体の変数に対し、ポインタを用いて書き込みを行う記述に対し、アプリケーションエラーが発生する問題を改修しました。

[発生条件]

V3.21よりも前のバージョンでは、以下の条件を満たす場合に本不具合が発生します。

① /Odオプションが指定されている。

② 構造体または共用体の変数SUVARが宣言されている。

③ SUVARは別の構造体または共用体の配列ARR_MEMBをメンバに持つ。

④ ポインタ変数PVAR がインデックス変数を用いてARR_MEMB のアドレスとして定義されている。

⑤ PVAR のメンバに定数0が代入されている。

 

[発生例]

typedef struct {

    unsigned short *pBuf; 

    unsigned short buf_size;

    unsigned short ix; 

} tBuffer;

 

typedef struct {

    tBuffer Buf[3];

} tWork;

 

tWork gstWork;

void func( unsigned char chNo )

{

    tBuffer *pstBuf;

 

    pstBuf   = &gstWork.Buf[chNo];

    pstBuf->ix  = 0;

}

 

(8)  8ビット/16ビット単位の書き込みにおいて、書き込み先の領域よりも大きいサイズで書き込みが行われる問題を改修しました。

[発生条件]

V3.21よりも前のバージョンでは、以下の条件を満たす場合に本不具合が発生します。

① コンパイル時に/Od以外の最適化オプションを指定している。

② farポインタを引数に持つ関数があり、その引数がレジスタに割り付けられている。

③ farポインタ引数を参照する式がある。

④ ③の後に以下のいずれかの処理を行っている。

・farポインタ引数をsigned short/unsigned short/signed int/unsigned int/signed long/unsigned long型のいずれかの型にキャストし、0xFFのAND演算を行った結果をsigned char型またはunsigned char型の変数に代入している。

・farポインタ引数をsigned long/unsigned long型のいずれかの型にキャストし、0xFFFFのAND演算を行った結果をsigned short/unsigned short/signed int/unsigned int型のいずれかの型の変数に代入している。

 

[発生例]

unsigned char var;

unsigned short __far *fp;

void fn( unsigned short __far *addr)

{

    fp = addr;

    var = ( unsigned char )( ( unsigned short )addr & 0xFF );

    return ;

}

[発生条件]

② signed long/unsigned long型のいずれかの型をfarポインタにキャストした結果を返す関数がある。

③ ②の関数を#pragma inlineを使用してインライン展開している。

④ ③のインライン関数の結果を用いてデータのアクセスを行っている。

 

[発生例]

{

}

    var = *(set_far_addr(0x10040)+i);

}

4. 制限事項

現バージョンのCCU8には、以下の制限事項があります。

(1)ネスト関係にあるインライン関数内にて、スタックに割り付けられたローカル変数のアクセスに対するコード生成を誤ります。

[発生条件]

① ネスト関係にあるインライン展開対象の関数が存在する。

ここでは、呼び出し側のインライン関数をinlinefunc1、inlinefunc1から呼び出されるインライン関数をinlinefunc2とする。

② inlinefunc2に対する呼び出しが以下のいずれかを満たす。

(a) inlinefunc2の呼び出しが、inlinefunc1関数のループ内に存在する。

(b) 戻り値を持たないinlinefunc2に対して、inlinefunc1からの呼び出しが2回以上ある。

③ インライン関数内の記述が以下のいずれかを満たす。

(a) inlinefunc1にvolatile修飾されたローカル変数が2個以上存在し、そのローカル変数をinlinefunc2呼び出しの引数に2個以上指定してinlinefunc2にてその引数をアクセスしている。

(b) inlinefunc1のvolatile修飾されたローカル変数をinlinefunc2呼び出しの引数に1個以上指定し、inlinefunc2にてその引数とinlinefunc2のvolatile修飾されたローカル変数を1個以上アクセスしている。

(c) inlinefunc1のvolatile修飾されたローカル変数１個以上を同一ブロック内のinlinefunc2呼び出しの前でアクセスし、inlinefunc2にてinlinefunc2のvolatile修飾されたローカル変数を1個以上アクセスしている。

④ コンパイル時に最適化を有効にしている。

IDEU8では以下のいずれかに該当する。カッコ内はコマンドラインオプション。

- 基本的な最適化のみ　（指定なし）

- ループに関する最適化を行い、広域的な最適化を行わない　（/Ol）

- 広域的な最適化を行い、ループに関する最適化を行わない　（/Og）

- 広域的な最適化とループに関する最適化を行う　（/Og /Ol）

- 最適化を繰り返して、最大限に最適化する　（/Om）

- 実行速度の最適化　（/Ot）

 

[発生例]

#pragma inline inlinefunc1

#pragma inline inlinefunc2

 

int gvar1;

 

void inlinefunc2( int a, int b )

{

    volatile int lVar;

    lVar = a + b;

    gvar1 += lVar;

}

 

void inlinefunc1(void)

{

    volatile int a = 20;

    int b = 40;

    int index;

 

    index = 0;

    for(index =0; index <= 1; index++)

    {

        inlinefunc2( a, b );

    }

}

 

void func(void)

{

    inlinefunc1();

}

 

[回避策]

呼び出し側のインライン関数（例ではinlinefunc1）に対するinlineプラグマを外してください。

【例】

//#pragma inline inlinefunc1  // inlinefunc1に対するinlineプラグマを削除

#pragma inline inlinefunc2

 

int gvar1;

 

void inlinefunc2( int a, int b )

{

    ...

}

 

void inlinefunc1(void)

{

    ...

    for(index =0; index <= 1; index++)

    {

        inlinefunc2( a, b );

    }

}

 

 

 (2) const変数をsegconstまたはabsoluteプラグマを用いて絶対番地に割り付けようとすると、インターナルエラーが発生する場合があります。

[発生条件]

① const修飾された構造体変数が初期化付で定義されている。

② ①の変数は、2つ以上のメンバを持ち、2番目以降に別の構造体のメンバのアドレスを初期値として与えている。

③ ①の変数をsegconstまたはabsoluteプラグマで絶対番地に割り付ける指定をしている。

 

[発生例]

typedef struct di{

    unsigned int d0;

    unsigned int addr;

}TTBL;

 

typedef struct{

    unsigned char volat_a[2];           

    unsigned char current_a[3];         

}TVAR;

 

TVAR Varible;            

 

#pragma absolute CnstTbl 1:0x1000

const TTBL __far CnstTbl[]=

{

    {

        0x00,

        (unsigned int)Varible.current_a,

    }

};

 

 

[回避策]

segconstプラグマを用いて名前付きで定義したセグメントにconst変数を割り付け、リンク時のオプションにて割り付けアドレスを指定してください。

【例】

#pragma segconst __far "table_seg"

const TTBL __far CnstTbl[]=

{

    {

        0x00,

        (unsigned int)Varible.current_a,

    }

};

　IDEU8の[ターゲットオプション]ダイアログの[一般]タブにて、「優先的に割付けるTABLEセグメント」をチェックし、「セグメント指定」フィールドに table_seg-1:1000h のように指定してください。

 

 

 

(3) 同一ファイル内に、関数名の先頭のアンダースコア有無だけが異なる関数が存在する場合に、コンパイラが不正なコードを生成する場合があります。

[発生条件]

① 同一ファイル内に、関数名の先頭のアンダースコアの有無だけが異なる関数が存在する。（それぞれの関数名をfuncAおよび_funcAとする）

② 関数funcAの内部でスタックに割り付けられる変数を使用している。

③ 関数_funcAは別の関数（funcBとする）から呼び出されている。

 

[発生例]

static int _funcA(int intReq)

{

    return 0;

}

 

int dummy(void);

int funcA(void)

{

    volatile int rc;

 

    rc = dummy();

    return rc;

}

 

void funcB(void)

{

    _funcA(0);

    return;

}

 

[回避策]

関数名の違いが先頭のアンダスコア有無だけにならないよう、関数名を変更してください。

【例】

static int xfuncA(int intReq)  // 関数名を_funcA→xfuncAに変更

{

    return 0;

}

 

int dummy(void);

int funcA(void)

{

    volatile int rc;

 

    rc = dummy();

    return rc;

}

 

void funcB(void)

{

    xfuncA(0);       // 関数名を_funcA→xfuncAに変更

    return;

}

 

 

 (4) シフト演算と関数呼び出しを含む記述に対し、アプリケーションエラーが発生する場合があります。

[発生条件]

① 最適化オプションで/Om, /Ot, /Ogのいずれかを指定している。

② 変数と定数のAND(&)演算の結果を変数に代入する式がある。

③ ②の後に、代入された変数をビット幅とするシフト演算がある。

④ ③の後に、関数呼び出しがある。

⑤ ④の後に、変数と定数のAND(&)演算の結果を変数に代入する式がある。

⑥ ⑤の後に、代入された変数をビット幅とするシフト演算がある。

⑦ ②と⑤のAND(&)演算で使用される定数は同じ値である。

 

[発生例]

void fn();

unsigned char bit_pos;

unsigned char tmp;

unsigned char tmp2;

 

static void test(unsigned char arg1)

{

    bit_pos = arg1 & 0x0f;

    tmp = (tmp2 << bit_pos);

    fn();

    bit_pos = arg1 & 0x0f;

    tmp = (0x01 >> bit_pos);

}

 

[回避策]

不具合が発生する関数に対し、最適化プラグマでdefaultを指定してください。

【例】

  #pragma optimization default

  static void test(unsigned char arg1)

  {

      ...

  }

  #pragma optimization

 

(5) ワーニングW5013を出力すべき箇所に対し、W5013が出力されない場合があります。

[発生条件]

下記の①または②の条件を満たす場合に発生します。

①int/unsigned int/short/unsigned short/型の引数に対し、char/unsigned char型の値を指定している。

②char/unsigned char型の引数に対し、int/unsigned int/short/unsigned short型の値を指定している。

[発生例]

void func_uc(unsigned char x);

void main(void) {

    func_uc(0x100U); ・・・(A)

}

 

ワーニングW5013は、関数引数に対しバイト数の異なる型が指定されたために型の変換が行われたことを意味します。

上記(A)に対し、定数0x100Uはunsigned int型で、unsigned char型に変換されるため、W5013が出力されるべきですが、出力されません。

[回避策]

回避策はありません。

本ワーニングにつきましては出力されない場合があることをご留意の上参照ください。

 

(6) if-elseブロック内とその後ろに共通式があるような記述に対しアプリケーションエラーが発生する場合があります。

[発生条件]

① コンパイル時に最適化を有効にしている（/Om、/Ot、/Og、デフォルトのいずれかを指定）

② if-else内およびその後ろに論理ORを行う式が記述されている。

③ 上記②の論理ORの式はすべて同じである。

④ 上記②の論理ORのオペランドに、コンパイル時に値が0以外に確定するものが指定されている

　　（下記発生例では、変数Operandが該当。初期値１が設定されており、コンパイル時にはその値（1）が使用される）。

 [発生例]

int Test(unsigned int ResultVar, int CondnVar)

{

    unsigned int  Operand = 1;

    if (10 > CondnVar)

    {

       ResultVar = ResultVar || Operand;

    }

    else

    {

       ResultVar = ResultVar || Operand;

    }

    ResultVar = ResultVar || Operand;

    return ResultVar;

}

[回避策]

　共通式の中で使用されている変数をvolatile宣言してください。上の例では、変数Operandをvolatile宣言することで回避できます。

【例】

    volatile unsigned int  Operand = 1;

 

(7) コンパイラの最適化等の影響により、デバッガにて変数を正しく参照できない場合があります

CCU8コンパイラの最適化の影響によりローカル変数がレジスタに割り付けられた場合、ローカル変数に対するデバッグ情報の整合が取れなくなる場合があります。この場合、DTU8デバッガでその変数を正しく参照できなくなります。

 

(8) レジスタに割り付けられた関数引数のアドレスは取得できません

レジスタに割り付けられた関数引数のアドレスを取得することはできません。

int *p;

void func(int arg1)  ←引数arg1はレジスタに割り付けられます

{

    p = &arg1;  ←この行に対してエラー(E5009 : '&' on register variable)を出力します。

}

関数引数のアドレスを取得する必要がある場合には、関数を__noregで修飾してください。

__noregで修飾することにより、関数引数はすべてスタックに割り付けられるようになり、引数のアドレスを取得できるようになります。

void __noreg func(int arg1)

{

    p = &arg1;

}
